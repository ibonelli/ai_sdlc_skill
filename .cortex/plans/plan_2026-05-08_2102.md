### Step 2: Write the main `SKILL.md` (entry point)

This file will contain:
- **Frontmatter**: `name: sdlc`, description with trigger phrases ("SDLC", "project planning", "scoping", "requirements", etc.)
- **Workflow Step 1 — Progress Detection**: Scan `docs/` folder for existing phase artifacts, build a progress map (Phase 0: complete, Phase 1: partial, Phase 2: not started, etc.)
- **Workflow Step 2 — Present Menu**: Show progress status and offer:
  - "Continue from next incomplete phase" (sequential mode)
  - "Jump to a specific phase" (pick mode)
  - "Start a new project from Phase 0" (fresh start)
- **Workflow Step 3 — Route**: Load the appropriate `phase-X-*/SKILL.md`
- **ADR Proactive Check**: After any phase completes, ask "Did any decisions in this phase warrant an ADR?" → if yes, load `references/adr-template.md` and generate

### Step 3: Write each phase sub-skill (7 files)

Each phase sub-skill follows the same structure (~100-150 lines each):
1. **Frontmatter** with `parent_skill: sdlc`
2. **Prerequisites**: Which prior artifacts must exist (auto-read them)
3. **Batch Questions**: All questions to ask the user for this phase (presented together via `ask_user_question`)
4. **Generation**: Load template from `references/templates.md`, fill with user answers + prior phase context
5. **Validation**: Cross-reference generated content against prior phase artifacts (e.g., Requirements must trace to Problem Statement)
6. **Write & Confirm**: Write artifacts to disk, warn about any skipped sections
7. **ADR Check**: Proactively ask if any decisions need an ADR
8. **Next Phase**: Offer to continue to next phase or stop

**Phase-specific question batches:**
- **Phase 0**: Project name, problem description, stakeholders, scope boundaries, success metrics, known constraints, risks
- **Phase 1**: Personas/users, functional requirements, NFRs (security/perf/availability), data requirements, integrations, acceptance criteria
- **Phase 2**: Milestones, work breakdown, dependencies, risks/mitigations, rollout strategy, observability needs
- **Phase 3**: Service boundaries, interfaces/APIs, data entities/events, schemas, security model, deployment model
- **Phase 4**: Files touched, scope implemented, edge cases, limitations, migration steps, local testing instructions
- **Phase 5**: Test types needed, environments, entry/exit criteria, test cases per requirement
- **Phase 6**: System overview details, alert definitions, common tasks, onboarding steps
